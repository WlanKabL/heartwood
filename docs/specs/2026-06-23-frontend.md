# Frontend — Design

- **Date:** 2026-06-23
- **Status:** Approved, ready for implementation plan
- **Sub-project:** 2 of 3 (Multi-Tenant Backend → **Frontend** → Deployment)

## Goal

Replace the deliberately ugly server-rendered login/token page with a launchable, hosted
product surface: an editorial marketing site that explains Heartwood, a setup guide, a wiki
plus FAQ, GitHub login, API-token management, and the headline feature, a distinctive visual
tree viewer and editor. It must be hostable with `docker compose` on the founder's own server,
sit cleanly next to the existing Fastify backend, and not collide with the parallel V2
backend work happening on `main`.

## Constraints and context

- Backend is already multi-tenant (Postgres, Drizzle, Fastify): GitHub-OAuth browser session,
  per-user API tokens, MCP over Streamable HTTP at `/mcp`, and `GET /trees/:treeId/roots`.
- A second agent is implementing V2 on `main` (trust fixes, `list_trees`, `delete_tree`,
  `search_truths`, volatility warnings, a Stop hook). Those are **MCP tools and core changes**,
  not REST. This sub-project must not edit the files that agent owns
  (`core/hardness.ts`, `core/types.ts`, `core/tree.ts`, `core/create.ts`, `core/write.ts`,
  `core/service.ts`, `core/repository.ts`, `storage/postgres-trees.ts`, `mcp/server.ts`,
  `bin/stop-hook.mjs`, `docs/usage.md`). We **import** from the core, we do not modify it.
- UI language is **English only** (no i18n). Strict TypeScript, named exports, async/await,
  Tailwind classes only, no `any`.

## Design direction (locked)

A warm, editorial "field guide / specimen plate" language, deliberately the opposite of the
generic AI-dashboard look.

- **Tone:** warm paper as the primary theme (cream `#efe6d6`, ink `#2a2018`, rust/amber accent,
  a walnut hardness ramp), warm dark (espresso/bark, not blue-black) as the dark-mode switch.
- **Type:** Fraunces (serif display, editorial), Inter (body), JetBrains Mono (labels, data,
  technical detail).
- **Devices:** mono masthead, chapter numbers, a table-of-contents index, footnotes, page
  numbers, hairline baseline rules, real macro wood photography full-bleed, asymmetric layouts
  with deliberate overlap. No centered hero stacks, no stat-tile rows, no gradient glow blobs.
- **Signature object:** a tree-trunk cross-section (growth rings). The rings double as the
  brand motif **and** the live tree visualization: dense dark core = roots / high hardness,
  pale outer wood = leaves / volatile. Hardness reads as ring density plus an amber→walnut
  ramp; hairline callouts annotate real node values.

The reference for quality and warmth is the founder's KeeperLog site; Heartwood is a sibling
in the portfolio with its own identity (rings, not reptiles).

## Stack and hosting

- **Nuxt 4** in a new top-level `web/` directory, fully decoupled from `src/`.
- Hybrid rendering: marketing, docs and wiki routes are **prerendered** (`nuxt generate`,
  good SEO); the authenticated app routes are **client-rendered** (SPA) with a `200.html`
  fallback.
- **Tailwind** with custom design tokens (the palette and fonts above) mapped to the real
  project tokens. d3-hierarchy and d3-shape for the layout math of the visualization; the
  SVG itself is hand-rendered in Vue, no generic chart library (that would re-introduce the
  AI-generic look).
- Light state via Nuxt composables and `$fetch`; no heavy store unless the editor needs it.
- **nginx** serves the static build and reverse-proxies `/api`, `/auth`, `/mcp`, `/trees`
  to the backend, so the whole app is same-origin and needs no CORS. Session cookies work
  unchanged.
- `docker compose` brings up three services: `db` (existing), `app` (the Fastify backend,
  via a new thin backend `Dockerfile` that does not touch backend source), and `web` (nginx
  + the built frontend). `PUBLIC_URL` is the public origin (the nginx front door); the GitHub
  OAuth callback stays `${PUBLIC_URL}/auth/github/callback`.

## Information architecture

**Public (prerendered):**

- `/` — landing in the field-guide style: hero ("Give your agents a spine"), the problem
  (forgetful agents), the model (roots → trunk → branches → leaves), hardness explained, the
  iron rule (AI proposes, server enforces), connect-in-two-minutes, the dogfood proof
  (KeeperLog, ZentraX), footer with GitHub and portfolio links.
- `/docs` — setup guide: sign in, mint a token, add `.mcp.json`, add the SessionStart hook,
  build the first tree. Copy-paste blocks throughout.
- `/wiki` — field notes: the concept in depth plus an FAQ (what hardness is, why the AI may
  not re-rate, what a cascade is, why the friction is the feature).

**App (authenticated, client-rendered):**

- `/app` — trees overview: the user's trees with node counts; create a tree (name plus first
  root truth) and delete a tree.
- `/app/trees/:treeId` — the visualization plus editor (see below). The headline feature.
- `/app/tokens` — API-token management (list, create-once reveal, revoke). Replaces the old
  server-rendered page, in the new design language.
- `/app/settings` — minimal: profile and logout.

## Authentication

The browser uses the **existing cookie session** (GitHub OAuth, already implemented). All SPA
data calls are same-origin and authenticate via that session cookie, **not** a bearer token.
API tokens remain exclusively for the MCP / Claude Code connection.

One small backend touch, outside the V2 agent's file set: after the OAuth callback,
`completeGithubLogin` redirects to `/app` instead of `/`, so a human lands in the app. The old
`src/http/pages.ts` is left in place (still tested, harmless); nginx simply never routes the
public path to it.

## REST API (new adapter, `src/http/tree-routes.ts`)

A thin, session-authed adapter that reuses the **same core use-cases the MCP tools call**, and
returns whatever the core returns (so the V2 agent's additive fields — cascade preview,
`hardnessNote`, `volatilityWarning`, `similarTo` — flow through as JSON automatically). It
imports core functions; it never edits them. Registered with one line in `src/http/server.ts`,
scoped per request via `treeStore.forUser(userId)` where `userId` comes from the session.

| Method | Path | Core use-case | Notes |
| --- | --- | --- | --- |
| GET | `/api/trees` | list summaries | tree id plus node count |
| DELETE | `/api/trees/:treeId` | delete tree | `confirm` query; preview without it |
| GET | `/api/trees/:treeId` | `getResolvedTree` | nested forest, hardness, protected |
| GET | `/api/trees/:treeId/search?q=` | search nodes | for large trees |
| POST | `/api/trees/:treeId/nodes` | `createNode` | `parentId:null` = new root |
| PATCH | `/api/trees/:treeId/nodes/:nodeId` | `updateNode` | `confirm` for protected |
| POST | `/api/trees/:treeId/nodes/:nodeId/move` | `moveNode` | `newParentId`, `confirm` |
| DELETE | `/api/trees/:treeId/nodes/:nodeId` | `deleteNode` | `confirm` |

Reused as-is: `GET /api/me`, `GET/POST/DELETE /api/tokens`.

**Dependency note (list and search):** if the V2 agent's `listTreeSummaries` and `searchNodes`
repository methods are already on `main`, the adapter calls them. If not yet, the adapter
performs the equivalent tenant-scoped Drizzle query directly against `deps.db` (the same
pattern `src/http/api-routes.ts` already uses for tokens), so this sub-project is never blocked
on the other agent. It still never edits core files.

## The visualization and editor (the heart)

`/app/trees/:treeId` opens on the **ring view** as the signature overview, with a one-click
switch to an **outline view** for dense editing. Both are first-class and share one detail
panel and one set of edit actions.

- **Ring view:** a concentric radial layout. The protected core sits at the center, depth
  pushes outward; a forest of several roots occupies angular sectors. Hardness is encoded by
  the amber→walnut ramp plus a number badge; protected nodes carry a marker. This is the
  "fancy" hero blick and reuses the brand's cross-section motif. Layout via d3-hierarchy
  radial; rendering hand-built SVG. The outline view is the fallback for very dense trees.
- **Outline / structure view:** an indented, precise list (B's tech precision) that is the
  day-to-day editing workhorse and scales past what a radial can show cleanly.
- **Detail panel** (either view): `content`, `label`, hardness with its breakdown (set /
  structural / proven), `depthFromRoot`, `protected`, children.
- **Editing:** create a child, edit content/label/proposed hardness, move (re-parent), delete.
  A change to a **protected** node (hardness ≥ 60) triggers a **cascade-preview modal** ("this
  change invalidates these N descendants, confirm"), driven by the preview the REST call
  returns when `confirm` is absent; confirming re-sends with `confirm: true`. The friction is
  intentional, surfaced visibly, never silent.
- **Search** to jump within large trees.
- **Empty state:** a tree with no nodes invites the first root, echoing the marketing hero.

## Testing

- Frontend: component tests for the visualization layout math (depth → radius, hardness →
  color), the cascade-preview flow (preview shown, confirm re-sends), and token create/revoke.
  An e2e smoke (Playwright is available) for login → app → view a tree.
- REST adapter: route tests through the session-authed path, including tenant isolation (user
  B cannot read or write user A's tree via any `tree-routes` endpoint), mirroring the existing
  `src/http/isolation.test.ts` discipline. Happy path, failure path (401 unauthenticated, 404
  missing), and the protected-node confirm path.

## Build order

1. Scaffold `web/` (Nuxt 4, Tailwind tokens, fonts) plus the REST read API (`GET /api/trees`,
   `GET /api/trees/:treeId`) and the landing page.
2. Login wiring (OAuth redirect to `/app`) and the token-management screen.
3. Tree viewer: ring view and outline view with the detail panel (read-only first).
4. Editor: create, edit, move, delete, with the cascade-preview modal.
5. Docs, wiki and FAQ.
6. `web/Dockerfile`, nginx config, backend `Dockerfile`, and the extended `docker-compose.yml`.

## Out of scope

- Password and magic-link login (GitHub only, as in the backend).
- Any change to the hardness engine, MCP tools, or core repository (owned by the V2 agent).
- The deployment hardening / server provisioning beyond a working `docker compose` (that is
  sub-project 3).
- Public or shared read-only trees (a later phase 5 idea).
